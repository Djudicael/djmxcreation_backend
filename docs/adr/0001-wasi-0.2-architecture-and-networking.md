# 1. WASI 0.2 Architecture and Networking Strategy

Date: 2024-05

## Status
Accepted

## Context
We are migrating a standard multi-threaded Tokio/Axum Rust backend to run as a WebAssembly component using the modern **WASI 0.2 (Preview 2)** standard (`wasm32-wasip2`). The goal is to deploy the application to a custom Wasm Cloud Platform (Wasmer/Pingora).

Standard Rust networking code frequently fails to compile or run in WebAssembly for three main reasons:
1. **OS-Level Assumptions**: Libraries like `tokio-postgres` and `home` attempt to set advanced TCP socket flags (like `TCP_NODELAY` or keepalives) or look up local OS usernames (via `whoami` or `getpwuid_r`). These concepts do not exist in the Wasm sandbox and cause runtime panics.
2. **Browser Assumptions**: Many high-level HTTP clients (like `reqwest`) assume that any `wasm32` compilation target is a web browser. They automatically swap their TCP backends for `js-sys` and `wasm-bindgen` wrappers around the browser's `fetch` API. These Javascript futures are `!Send`, completely breaking async multithreading (`Send` trait bounds) in standard Tokio applications.
3. **Cryptography Native Dependencies**: Standard TLS libraries (like `ring` or `rustls`) heavily rely on native C compilers and assembly instructions, making them extremely difficult to compile to WebAssembly.

## Decisions

To build a robust, future-proof WASI 0.2 component without compromising our native development experience, we have made the following architectural decisions:

### 1. Enable Tokio's WASI 0.2 Socket Support
We are compiling the project with `RUSTFLAGS="--cfg tokio_unstable"`. This unlocks Tokio's bleeding-edge support for mapping standard Rust `TcpStream` operations to native WASI 0.2 sockets (`wasi:sockets/tcp`). We codified this inside `.cargo/config.toml` to automate the build pipeline.

### 2. Fork `tokio-postgres` and `home` (Local Patches)
To communicate with PostgreSQL over Tokio's WASI sockets, we maintain minimal local patches for `tokio-postgres` and `home` in our workspace (`[patch.crates-io]`).
* The `tokio-postgres` patch strips out `socket2` keepalive configuration.
* The `home` patch bypasses Linux `.pgpass` username lookups.
* We configure the database pool with `NoTls`, ensuring that unencrypted wire protocol data is sent over the WASI socket (deferring the actual network encryption to the Wasm Host/VPC layer).

### 3. Replace `rust-s3` with `aws-sdk-s3`
We abandoned the `rust-s3` crate because it hardcodes a dependency on `reqwest` and `tokio-rustls`. Instead, we adopted the official `aws-sdk-s3`.
* We use conditional compilation (`cfg(not(target_os = "wasi"))`) to use standard `hyper` and `rustls` for native Linux/Windows builds.
* For WASI builds, we include the SDK with `default-features = false` to strip out all TCP and TLS dependencies.

### 4. Custom WASI HTTP Connector (`WasiHttpClient`)
Because we stripped the HTTP execution engine from the AWS SDK on WASI, we implemented a custom `WasiHttpClient` (in `crates/repository/src/config/wasi_http_client.rs`). This connector implements the AWS `HttpConnector` trait and acts as a translation layer:
* It intercepts the AWS HTTP request.
* It maps the payload to `wasi::http::types::OutgoingRequest`.
* It delegates the actual HTTP execution and TLS negotiation to the WebAssembly Host (Pingora/Wasmer) using `wasi::http::outgoing_handler::handle`.
* It polls the host for the response and maps it back into an AWS Smithy response.

### 5. Removed Legacy `wasix-org` Ecosystem Patches
By completely removing `reqwest` and relying on host-provided `wasi:http` bindings, we no longer need to compile native TLS inside the Wasm module. Consequently, we removed all legacy `wasix-org` overrides for `ring`, `rustls`, and `rustls-native-certs` from our `Cargo.toml`.

## Consequences

### Positive
* **Tiny Wasm Binaries**: By deferring HTTP execution and TLS to the Wasm Host layer, our compiled `.wasm` components are significantly smaller and faster.
* **Native Parity**: Because we use conditional compilation interfaces (like AWS SDK's `HttpClient` trait), developers can still run `cargo run` natively on Windows/macOS/Linux using standard multi-threaded Tokio and Hyper, preserving a fast local development loop.
* **No `!Send` Panics**: We are entirely free from browser-based `js-sys` WebAssembly assumptions.

### Negative / Technical Debt
* We are currently responsible for maintaining the local forks of `tokio-postgres` and `home` in `crates/rust-postgres` and `crates/home-patch`. Once the upstream crates officially support WASI 0.2 without panicking on socket options, we should remove these patches.
* The `WasiHttpClient` adapter relies on raw `wasi:http` bindings. If the WASI standard evolves (e.g., Preview 3), this adapter will need to be updated.
